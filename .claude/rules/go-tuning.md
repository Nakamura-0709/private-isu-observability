# Go Tuning Rules

## N+1問題の検出と対処

1. `otelsql` のスパンを確認し、同一リクエスト内で同じクエリが繰り返されていないか確認する。
2. N+1が検出された場合は以下の順で対処を検討する。
   - クエリを `IN` 句でまとめてバッチ取得する
   - アプリケーション内キャッシュ（`sync.Map` や `map`）で重複クエリを削減する
   - 結合クエリで1回のDBアクセスにまとめる

## OpenTelemetry の計装

- 新しい関数を追加する際は `otelchi` または手動の `tracer.Start()` でスパンを計装すること。
- スパン名はHTTPメソッド+パス形式（例: `GET /posts/{id}`）に統一する。
- DBアクセスは `otelsql` でラップされたコネクションを使用し、クエリ単位でトレースを取得すること。

## キャッシュ導入の基準

- `pt-query-digest` で `Rows_examined / Rows_sent` 比が高いクエリを対象とする。
- キャッシュ導入前後で `make bench` のスコアを比較し、効果を定量化すること。
- キャッシュの有効期限はベンチマーカーの整合性チェックを通過できる範囲で設定する。
